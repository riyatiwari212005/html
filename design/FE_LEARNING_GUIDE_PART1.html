🍕Paper 1: Building Riviera - Declarative Real-Time Feature Engineering (DoorDash 2021)

Company: DoorDash | Year: 2021 | Domain: Feature Framework Declarative Auto-Generation Deduplication
⚙️ The Problem: Feature Explosion & Fragmentation

Scale of the Problem at DoorDash

By 2021, DoorDash had accumulated: 1,250+ features across 4 ML models with massive inconsistency and waste.

Feature Breakdown:

  • Delivery time prediction: 300+ features (manual code)
  • Demand forecasting: 250+ features (40+ duplicates!)
  • Merchant ranking: 400+ features
  • Customer recommendations: 300+ features

Debugging features 20% Finding which feature broke
Maintaining features 20% Ownership unclear

💡 Key Problem: Feature engineering takes 2-4 weeks per feature despite many overlaps. Inconsistency causes training-serving skew. Duplication wastes compute (merchant_rating computed 4 times!).
💡 Solution: Riviera Declarative Framework

Core Innovation: From Imperative to Declarative

Traditional Approach (Imperative):

# Developer writes code for each feature def compute_merchant_rating_7d(merchant_id): result = spark.sql(""" SELECT AVG(rating) FROM ratings WHERE merchant_id = ? AND timestamp > NOW() - 7 days """) return result # Different teams write it differently → inconsistency!

Riviera Approach (Declarative):

# Developer writes specification (not code!) merchant_rating_7d: entity: merchant_id source: ratings_table aggregation: mean aggregation_field: rating window: 7d filters: - rating > 0 # Riviera generates execution automatically! # Same specification → consistent everywhere

Key Features of Riviera:

  • YAML-based definitions: Human-readable, version-controllable
  • Automatic code generation: Spark SQL for batch, Flink for streaming
  • Dual-path execution: Batch + real-time from single definition
  • Built-in optimization: Deduplication, caching, sharing
  • Automatic testing: Validates all features for anomalies

🚗Paper 2: Optimal Feature Discovery - Information Theory Approach (Uber 2021)

Company: Uber | Year: 2021 | Domain: Feature Selection Information Theory 98% Reduction
⚙️ The Problem: Feature Bloat & Redundancy

The Paradox: More Features = Lower Performance

Year-by-Year Feature Count vs. Model AUC:

Year Features Model AUC Observation
2018 2,000 0.820 Baseline
2019 4,000 (2x) 0.830 +5% AUC only!
2020 7,000 0.830 No change
2021 10,000+ 0.820 Actually declining!
💡 Root Cause: Most features are redundant! Feature count grew, but true information stayed constant. Multicollinearity confusing the model.

Redundancy Analysis:

  • Feature pairs with correlation > 0.95: 3,000 redundant pairs
  • Implication: Of 10,000 features, maybe only 5,000 carry unique information
  • Example: user_orders_total, user_orders_1yr, user_orders_6m, user_orders_3m, user_orders_1m all measure same signal (order history)
💡 Solution: Information Theory-Based Feature Selection

Information Theory Approach

Key Metrics:

  • Mutual Information (MI): How much does feature X tell us about target Y?
  • Redundancy: Do X and Z tell us the same thing? (Is MI(X,Z) high?)
  • Goal: Find features with high MI(feature, target) AND low MI(feature, other_features)

Greedy Selection Algorithm:

# Information theory-based feature selection selected = [] remaining = all_features # Rank all features by MI(feature, target) feature_mi = compute_mutual_information_all() # Greedy: maximize Information Gain (info not redundant) while len(selected) < target_size: best_feature = None best_gain = 0 for feat in remaining: # Information gain = # MI(feat, target) - redundancy_with_existing redundancy = 0 for sel_feat in selected: redundancy += MI(feat, sel_feat) gain = MI(feat, target) - redundancy/len(selected) if gain > best_gain: best_gain = gain best_feature = feat selected.append(best_feature) remaining.remove(best_feature) return selected

Example: Why It Works

Traditional importance-based selection (WRONG):

  • user_orders_total: importance 0.15 ✓ Keep
  • user_orders_1yr: importance 0.14 ✓ Keep
  • user_orders_6m: importance 0.13 ✓ Keep
  • user_orders_3m: importance 0.12 ✓ Keep
  • Problem: All 4 measure same signal (highly correlated, redundant)

Information theory selection (CORRECT):

  • user_orders_total: MI = 0.20 ✓ Keep (high info!)
  • user_orders_1yr: MI(given total) = 0.01 ✗ Skip (redundant)
  • user_orders_6m: MI(given total,1yr) = 0.005 ✗ Skip (redundant)
  • Result: Keep 1, discard 3 (cleaner signal)

Model AUC 0.830 0.841 +1.3% (better!)
Features 10,000 253 97.5% reduction
Inference latency 150ms 15ms 10x faster

Business Impact

  • Model quality: +1.3% AUC (significant for Uber scale)
  • Inference speed: 10x faster means lower latency for rides
  • Cost savings: At 100M queries/day, 10x speedup = ~$1M/day savings
  • Annual potential: $365M+ (if applied to 10% of inference)
💡 Key Insight: Feature importance ≠ Feature uniqueness. High-importance features can be collinear. Information theory finds truly complementary features → smaller models, better quality, faster inference.

🚙Paper 3: ML Feature Serving Infrastructure at Lyft (2021)

Company: Lyft | Year: 2021 | Domain: Real-Time Feature Serving Latency <5ms 99.99% Uptime
⚙️ The Problem: Real-Time Feature Fetching Latency

Strict Time Budget: <100ms for Entire Flow

Ride Request Flow (Lyft):

T=0ms: Request received (user taps "request ride") └─ Goal: Return matched driver in <100ms (feels instant!) │ ├─ T=0-10ms: Feature loading (HARD BUDGET!) │ ├─ Current driver_rating: ? │ ├─ Current driver_location: ? │ ├─ Passenger_rating: ? │ └─ Surge multiplier: ? │ ├─ T=10-50ms: Feature serving & formatting ├─ T=50-80ms: Model inference (predict driver match) ├─ T=80-90ms: Post-processing └─ T=90-100ms: Send response (driver selected!)

Bottleneck: Database is Too Slow

Traditional: Fetch from database ├─ Network to DB: 10-15ms ├─ Query execution: 5-10ms ├─ Network return: 10-15ms ├─ Deserialization: 1-2ms └─ Total: 26-42ms (exceeds 10ms budget!) Result: Feature fetching alone violates time budget. Solution: Cache features in Redis (in-memory storage)

With Redis Cache:

  • Network latency: 2-3ms (Redis nearby!)
  • Lookup: 1ms (hash table)
  • Deserialization: 1ms
  • Total: 4-5ms (within budget!)

💼Paper 4: Near Real-Time Features for Personalization (LinkedIn 2022)

Company: LinkedIn | Year: 2022 | Domain: Real-Time Personalization Dual-Path +15% Engagement
⚙️ The Problem: Freshness vs. Consistency Paradox

The Core Tension

Real-Time Personalization Needs BOTH:

  • Freshness: User just viewed article → system learns immediately → recommend similar (within seconds!)
  • Consistency: Training used feature X at version 1 → serving must use same version → no training-serving skew

The Problem:

Fresh but inconsistent (❌): Serve most recent features but computed differently → skew → bad predictions

Consistent but stale (❌): Serve same as training but batch (overnight) → can't do real-time personalization

Fresh AND consistent (✓ but hard!): How to compute batch + realtime identically?

💡 Solution: Declarative Dual-Path Execution

Single Definition, Two Execution Paths

Feature Definition (Abstract YAML):

features: user_articles_viewed_1h: entity: user_id event_source: article_view_events aggregation: count time_window: 1h filters: - view_duration > 5s

Path A - Batch (Offline):

Trigger: Hourly job Computation: Spark SQL Query: SELECT count(*) FROM views WHERE user_id = X AND time > NOW() - 1h Output: Stored in feature store Used for: Model training (consistent snapshot)

Path B - Real-Time (Online):

Trigger: Kafka stream (continuous) Computation: Flink windowed aggregation Logic: Increment counter for 1h window Output: Update Redis (instant!) Used for: Model serving (most recent!)
💡 Key Insight: Both paths compute the SAME metric. Same definition = identical logic. Only execution mechanism differs (batch vs stream). Result: Guaranteed consistency!

📚 Evolution of Feature Engineering (2021-2022)

By Year - Major Innovations:

  • DoorDash 2021: Declarative frameworks eliminate manual coding; 5-7x faster feature velocity
  • Uber 2021: Information theory finds non-redundant feature sets; 97.5% reduction while improving quality
  • Lyft 2021: Real-time serving architecture with <5ms latency across millions of features
  • LinkedIn 2022: Dual-path execution (batch + streaming) enables real-time personalization with training consistency

Key Lessons Learned:

  1. Declarative > Imperative: YAML descriptions beat hand-coded features for consistency
  2. Redundancy matters: 97% of features can be noise; information theory finds true signal
  3. Freshness hierarchy: Different features need different update frequencies (hot/warm/cold)
  4. Consistency critical: Training-serving skew kills model quality; use same code paths
  5. Scale changes everything: Feature serving at Lyft/LinkedIn scale requires multi-layer caching
  6. Cost-quality tradeoff: Real-time systems cost $2M+ but enable $100M+ in business impact

Future Directions:

  • Automated feature discovery (rather than manual definition)
  • Feature stores as standardized platforms (Riviera, Feathr, etc.)
  • Real-time + batch convergence (streaming all the way)
  • Feature governance and lineage tracking
  • Open-source standardization (Feathr OSS at LinkedIn)
