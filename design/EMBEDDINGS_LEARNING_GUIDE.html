🛒Paper 1: Vector Embeddings for Shopping Carts and Customers

Company: Sears Corporation | Year: 2017 | Type: E-Commerce Recommendations Exponential Family Three-Fold System Hybrid Recommendations

🎯 Core Concept 1: Exponential Family Embeddings

📌 Why Geometric Space Matters

Embeddings are most effective when learned in the right space. Different problems need different approaches.

Traditional vs Exponential Family Approach:
❌ Euclidean Distance
  • Two items similar if ||v₁ - v₂|| small
  • Works for: Continuous features
  • Problem: Doesn't model probability
  • Loses: Business metric alignment
✅ Exponential Family
  • P(item_j | item_i) ∝ exp(v_i · v_j)
  • Works for: Count data (shopping)
  • Benefit: Directly models co-purchase probability
  • Gain: Optimizes actual business metric
💡 Why It Matters: Can directly answer "What's P(user buys B | bought A)?" and optimize for the probability that matters to business!
Mathematical Framework:
P(y | x) = h(y) × exp(⟨φ(x), t(y)⟩ - A(φ(x))) For shopping context: ├─ h(y): Base measure (prior) ├─ φ(x): Sufficient statistics ├─ t(y): Feature encoding of target ├─ A(φ): Partition function (normalization) │ Result: Naturally scaled probabilities └─ Direct optimization for what matters!

🎯 Core Concept 2: Three-Fold Embedding System

📌 Why Single Embeddings Aren't Enough

Sears discovered that three complementary embeddings work better than any single representation.

The Three Embeddings:
1️⃣ Product Embeddings
  • What: Each SKU as vector
  • From: Co-purchase patterns
  • Result: Products cluster by type/price/occasion
  • Example: Hammers near wrenches & screwdrivers
2️⃣ Customer Embeddings
  • What: Each customer as vector
  • From: Purchase history & behavior
  • Result: Customers cluster by shopping behavior
  • Example: Budget vs premium customers separate
3️⃣ Cart/Trip Embeddings
  • What: Shopping trips as vectors
  • From: Items + sequence in trip
  • Result: Intent vectors captured
  • Example: "Holiday gifts" vs "Emergency repair"
🎯 Three-Fold Advantage
  • Hybrid: Combines all three signals
  • Cold-start: Graceful degradation
  • Complexity: Captures context + intent
  • Accuracy: 2-3% improvement over single embedding
Why Separate Cart Embeddings Matter: Same customer, different trips = different intent. Trip 1 needs emergency repair recommendations, Trip 2 needs holiday decorating. Single customer embedding misses this nuance!

    • 🏬Paper 2: Billion-Scale Commodity Embeddings

    Company: Alibaba | Year: 2018 | Type: Graph-Based Embeddings at Scale Knowledge Graphs Side Information 1B+ Products

    🎯 Core Concept 1: Graph Embeddings for E-Commerce

    📌 Why Regular Skip-Gram Isn't Sufficient

    E-commerce is fundamentally a GRAPH problem, not a sequence problem like NLP.

    The Graph Structure:
    Products ←→ Products (co-purchase relationships) ↓ Products ←→ Categories (hierarchical structure) ↓ Items ←→ Attributes (shared features) ↓ Users ←→ Items (interaction signals) Traditional embeddings (skip-gram): ├─ Only use: Direct customer interactions ├─ Problem: Sparse! (user clicks 1000 of 1B items) ├─ Cold-start: New items = zero interactions └─ Result: Poor coverage Graph embeddings: ├─ Propagate: Information through structure ├─ User A ≈ User B ≈ User C ├─ User C clicked Item X ├─ Infer: Items X might be relevant to User A ├─ Cold-start: New items discovered via categories/attributes └─ Result: Better coverage even with sparsity
    💡 Alibaba's Scale Challenge: 1B+ products, 500M+ users, but sparsity = each user interacts with 0.0001% of items! Graph structure is essential!

    🎯 Core Concept 2: Side Information Integration

    📌 Enriching Embeddings with Metadata

    Side Information Types at Alibaba:
    1️⃣ Categorical Data
    • Product category (electronics, clothing)
    • Brand name
    • Shipping region
    • Seller type
    2️⃣ Hierarchical Data
    • Category → Sub-category → Product type
    • Enables: "Similar to X in same category"
    • Propagates: Information up/down hierarchy
    • Benefit: Regularizes embeddings
    3️⃣ Attribute Data
    • Color (for clothing): Red, Blue, Green
    • Size, Material, Price range
    • Connects similar products
    • Enables: Cross-category recommendations
    4️⃣ Signal Multiplication
    • Without side info: 100 clicks on product X
    • With side info: 100 clicks + 10K from category + 500 from attributes
    • Result: 100→10.6K training signal!
    • Benefit: Regularization through shared structure
    🎯 Real Example: New product with 0 clicks. But same category has 10K clicks, same color/size links 500 items. New product immediately gets embedding from side information! No cold-start week!

    🐦Paper 3: Real-Time Personalization at Twitter Scale

    Company: Twitter | Year: 2018 | Type: Real-Time Embeddings Multi-Representation Graph-Based Users Microsecond Latency

    🎯 Core Concept 1: Real-Time Constraints

    📌 Why Traditional ML Doesn't Work at Twitter

    The Challenge:
    Twitter Volume: ├─ 145K tweets per minute (2.4K per second!) ├─ 500M+ monthly active users ├─ Real-time feed ranking requirement └─ Microsecond-scale latency budgets Traditional ML problems: ├─ Model training: Takes days ├─ Fresh data: Hours old by deployment ├─ Inference latency: Can't afford long processing ├─ Cold-start: New tweets have zero engagement └─ Context volatility: Trends shift minute-by-minute
    Why Embeddings Solve This:
    ✅ Pre-Computation
    • Embeddings computed offline per item
    • Inference = simple math (dot product)
    • Latency: Microseconds
    • On-the-fly: Can't recompute at query time
    ✅ Semantic Richness
    • Capture meaning without explicit features
    • Cold-start: Reasonable default from text/metadata
    • Ranking: Instead of all 145K/min tweets
    • Efficiency: Find best from top-100 (still fast)

    🎯 Core Concept 2: Multi-Representation Learning

    📌 Why Single Embeddings Fail for Tweets

    Tweet Complexity:
    A tweet contains: ├─ Text content (main signal) ├─ Author profile (who said it) ├─ Images/media (visual signals) ├─ Engagement patterns (likes, retweets, replies) ├─ Temporal signal (when posted) └─ Social graph (who follows author) Single embedding can't capture all dimensions!
    Twitter's Multi-View Solution:
    1️⃣ Text Embeddings
    • From: Tweet text content
    • Use: Content-based discovery
    • Semantic: What is tweet about?
    2️⃣ Engagement Embeddings
    • From: Tweet interaction patterns
    • Use: Real-time popularity ranking
    • Signal: Is tweet trending?
    3️⃣ Social Graph Embeddings
    • From: User follow relationships
    • Use: Personalized rankings
    • Signal: Do I follow author?
    4️⃣ Temporal Embeddings
    • From: Time patterns in engagement
    • Use: Time-sensitive ranking
    • Signal: Is this time-relevant?

    📊 Real-Time Inference Flow

    ┌──────────────────────────────────────────────────────┐ │ TWITTER REAL-TIME TIMELINE SYSTEM │ ├──────────────────────────────────────────────────────┤ │ │ │ T=0ms: User Opens Timeline │ │ └─ Input: user_id, timestamp, device context │ │ │ │ T=1ms: Candidate Fetching │ │ ├─ Query: Tweets from followed accounts (24h) │ │ ├─ Result: 10K+ candidate tweets │ │ └─ Data: Real-time feed service │ │ │ │ T=5ms: User Embedding Aggregation │ │ ├─ Query: Get recent likes (last 1000) │ │ ├─ Aggregate: Average tweet embeddings │ │ ├─ Decay: Weight older likes lower │ │ └─ Result: current_user_embedding │ │ │ │ T=10ms: Embedding-Based Retrieval │ │ ├─ For 10K candidates: │ │ ├─ Retrieve: Pre-computed embeddings │ │ ├─ Compute: cos(user_emb, tweet_emb) │ │ ├─ Keep: Top 500 by similarity │ │ └─ Result: Coarse stage candidates │ │ │ │ T=15ms: Deep Ranking (500 selected) │ │ ├─ Deep neural network scores all 500 │ │ ├─ Input: Embeddings + all features │ │ ├─ Output: Ranking scores │ │ └─ Sort: Final order │ │ │ │ T=20ms: Re-Ranking & Diversification │ │ ├─ Diversify: Not too many from one author │ │ ├─ Monetize: Insert ads │ │ ├─ Fairness: Apply constraints │ │ └─ Truncate: Top 30-50 tweets │ │ │ │ T=25ms: Return to User │ │ └─ HTTP response with final timeline │ │ │ │ T=125ms: Render (User sees timeline) │ │ └─ Network + visualization time │ │ │ │ TOTAL: 25ms system, 100ms+ network │ │ │ └──────────────────────────────────────────────────────┘

    🏠Paper 4: Listing Embeddings for Recommendations at Airbnb

    Company: Airbnb | Year: 2018 | Type: Booking-Driven Embeddings Global Context Market-Level Sampling Cold-Start Solution

    🎯 Core Concept 1: Booking as Ultimate Signal

    📌 Why Booking Beats Clicking

    The Signal Hierarchy:
    User searches: "NYC, June 1-5, 2 beds" ├─ Clicks: [L1, L2, L3, L4, L5] (5 listings) ├─ Browses: Scrolls through options ├─ Views: Reads reviews and images └─ Books: L3 (commits financially!) Traditional approach: ├─ Problem: All clicks treated equally ├─ L1 click = L3 click (even though L3 booked!) └─ Loss: Noisy signal, hard to distinguish preferences Airbnb's insight: ├─ Booking = true intent signal ├─ Multiple clicks but single booking ├─ Financial commitment = honest preference ├─ Directly optimizes business KPI (bookings) └─ Much stronger training signal!
    Skip-Gram Modification:
    Standard skip-gram on click sequence: [L1, L2, L3, ...] ├─ Problem: All treated equally └─ Loss: Missing booking signal Airbnb's approach - Global Context: ├─ Booked listing (L3) = special ├─ Add L3 to negative sampling with HIGH weight ├─ Effect: L3 embedding pulled strongly toward quality ├─ Result: Booked listings are special in space └─ Outcome: Better embeddings for recommendations!

    🎯 Core Concept 2: Market-Level Negative Sampling

    📌 Why Negatives Matter More Than Positives

    The Sampling Challenge:
    ❌ Random Negative Sampling
    • Positive: NYC listings booked
    • Negatives: ANY listings (anywhere!)
    • Problem: NYC ≠ Paris (different market)
    • Issue: Paris apartment is not competitive
    ✅ Market-Level Negatives
    • Positive: NYC listings booked
    • Negatives: NYC listings NOT booked
    • Benefit: Only relevant competitors
    • Result: NYC embeddings cluster tight
    Example: User booked luxury apartment in Manhattan. Negative samples should be other Manhattan apartments (similar market, price point), NOT budget backpacker hostels (different market entirely). Market-level sampling learns precise similarities!

    🎯 Core Concept 3: Cold-Start Solution

    📌 Geometric Proximity for New Listings

    The Problem:
    New listing added to Airbnb: ├─ Zero booking history ├─ Can't use booking-based training ├─ Need immediate embedding └─ Traditional approach: Random or content-only (poor!)
    Airbnb's Cold-Start Solution:
    For new listing L_new: 1. Find 3 nearest neighbors: ├─ Geographic proximity: Same neighborhood ├─ Category match: Same bedrooms/type └─ Price range: Within 2x of L_new 2. Cold-start embedding: └─ emb(L_new) = [emb(L_near1) + emb(L_near2) + emb(L_near3)] / 3 3. Why this works: ├─ Nearby listings = similar market ├─ Same type = similar target customer ├─ Similar price = similar segment └─ Average starts in "right part" of space 4. Improvement over time: ├─ As bookings arrive: Embedding pulled from cold-start ├─ Quickly converges: After 10-20 bookings ├─ Eventually: Same quality as established listing └─ No week-long cold period!
    💡 Result: Cold-start embeddings ~80-90% as good as fully trained! New listings discoverable immediately instead of being invisible for weeks!

    • 👗Paper 5: Latent Style Embeddings via Implicit User Feedback

    Company: Stitch Fix | Year: 2018 | Type: Fashion Style Embeddings Latent Space Implicit Feedback Keep-Rate Optimization

    🎯 Core Concept 1: Style as Latent Representation

    📌 Why Fashion is Different

    Fashion Challenge:
    What makes outfit "good" for person X? ├─ Hard to describe: "I like it" vs "I can't explain why" ├─ Subjective: Personal preference, not objective ├─ Traditional features fail: Color names ambiguous ├─ Style is multi-dimensional AND personal └─ Aesthetic changes: Trends shift rapidly
    Latent Style Approach:
    Hypothesis: Style exists in latent space Observable: Keep/return decisions ├─ Keep: Item aligned with person's style ├─ Return: Item misaligned with style └─ These decisions reveal preferences! Model: ├─ Person has style vector: s_person (unknown) ├─ Item has style vector: s_item (unknown) ├─ P(keep | person, item) ∝ cos(s_person, s_item) │ ├─ Training: Observe keep/return data ├─ Learn: Both person and item vectors ├─ Result: Shared style space └─ Interpretation: Similar people/items cluster!
    💡 Unique Advantage: Stitch Fix has explicit keep/return signal. Not implicit like clicks - it's actual purchase/return behavior with money backing it!

    🎯 Core Concept 2: Implicit User Feedback

    📌 Why Return Behavior is Powerful

    Feedback Types:
    ❌ Explicit Feedback (Not Used)
    • Ratings: "Rate this 5 stars"
    • Problem: Users rarely rate
    • Sparse: Very few ratings available
    • Biased: People rate extremes
    ✅ Implicit Feedback (Stitch Fix)
    • Keep: Strong signal "I like this"
    • Return: Strong signal "Doesn't work"
    • Money: Financial commitment backing
    • Natural: No extra effort from customer
    Why Implicit is Better: ├─ Natural: No extra effort needed ├─ Honest: Real money backing preferences ├─ Aligned: Matches business metric (keep rate) └─ Rich: Keep vs return both informative!

    📌 LinkedIn's Unique Advantage

    The Economic Graph Structure:
    Professional Network (200M+ members): Nodes: ├─ Professionals (members) ├─ Companies (employers) ├─ Schools (education) ├─ Skills (professional capabilities) ├─ Roles (job titles/positions) ├─ Locations (geography) └─ Industries (sectors) Edges: ├─ Connection: Direct follows ├─ Employment: Works at company ├─ Education: Attended school ├─ Endorsement: Skill endorsed ├─ Role: Currently held position └─ Experience: Similar background
    💡 Unique Data Asset: Only LinkedIn has this complete professional relationship graph. 200M professionals with rich verified employment history. This is the foundation for talent matching!
    Why This Matters for Talent Search:
    Traditional recruiting: ├─ Recruiter searches: "Software Engineer in NYC" ├─ System matches: Title + Location + Skills ├─ Problem: Exact keyword match misses great candidates ├─ Example: "Principal Engineer" doesn't match "Sr. SWE" └─ Result: Perfect fits might be invisible LinkedIn Graph-Based: ├─ Candidate A: Principal Engineer (slightly diff title) ├─ But same company, same skills as target profile ├─ And worked with similar hiring manager before ├─ Graph propagates: This person is similar! └─ Result: Great candidate discovered!

    🎯 Core Concept 2: Sparse Entity Embeddings

    📌 Managing the Curse of Dimensionality

    The Challenge:
    Entity types at LinkedIn: ├─ Users: 200M+ ├─ Companies: 1M+ ├─ Schools: 50K+ ├─ Skills: 100K+ ├─ Locations: 10K+ └─ Roles: 1000+ Naive one-hot encoding: ├─ Total dimensions: 200M + 1M + 50K + 100K + 10K + 1K ├─ = 201 MILLION dimensions! ├─ Too high, requires massive memory ├─ Training would be prohibitively slow └─ Most users have only 1-2 of each type
    Sparse Entity Embeddings Solution:
    Instead of one-hot (1M dims), use sparse embeddings: For each entity type: ├─ Company: 1M entities → 20D embedding ├─ School: 50K entities → 20D embedding ├─ Skills: 100K entities → 20D embedding (per skill!) ├─ Role: 1K entities → 20D embedding ├─ Location: 10K entities → 20D embedding └─ Total: ~160D (not 201M!) Training efficiency: ├─ Only updates non-zero values ├─ Memory: Manageable in-memory ├─ Speed: Fast gradient computation └─ Scalability: Trains on modern hardware Concatenation: ├─ Company emb: 20D ├─ School emb: 20D ├─ Skills embs: 100D (multiple skills summed) ├─ Role emb: 20D └─ Total candidate embedding: 160D

    📌 Core Insights Across All Papers

    1. Choose the Right Space

    Different problems need different embedding spaces. Exponential family for probabilities, graphs for networks, latent space for subjective preferences.

    2. Multiple Perspectives

    Single embeddings often insufficient. Combine product + customer + context. Use text + engagement + social + temporal signals.

    3. Signal Quality Matters

    Implicit feedback from real actions beats explicit ratings. Bookings > clicks > views. Keep/return > likes > follows.

    4. Context is Everything

    Market-level negatives, geographic proximity for cold-start, recruiter preferences for ranking. Context improves precision.

    5. Cold-Start Solutions

    Side information, geometric proximity, metadata-based initialization. Solve cold-start immediately instead of week-long ramp-up.

    6. Scalability Through Efficiency

    Sparse embeddings, distributed processing, approximate algorithms. Handle billions at millisecond latency.
