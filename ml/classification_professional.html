When to Use StandardScaler?
Algorithm Requires Scaling Why
Logistic Regression ✓ YES Distance-based gradient descent
SVM ✓ YES Kernel computation sensitive to scale
Decision Tree ✗ NO Splits based on values, not distances
Random Forest ✗ NO Ensemble of trees, scale-invariant
XGBoost ✗ NO Handles scaling internally

Pro Tips & Best Practices

Production Deployment Tips

  • Separate functions for data loading, preprocessing, training, evaluation
  • Use config objects or dictionaries for hyperparameters
  • Write docstrings for all functions
  • Make code reusable across different datasets

  • Wrap external API calls in try-except blocks
  • Implement smart fallback mechanisms
  • Log warnings and errors for debugging
  • Check for missing files before processing

  • Always set experiment name for organization
  • Log hyperparameters BEFORE training
  • Log metrics after training with consistent names
  • Save model artifacts with tags
  • Run mlflow ui to compare experiments

  • Save models as joblib (smaller than pickle)
  • Save preprocessing objects alongside model
  • Document feature names and order
  • Version models with timestamps
  • Create metadata JSON with model info

  • Use stratified train/test split to preserve distribution
  • Use F1-score (macro) instead of accuracy
  • Consider SMOTE for severe imbalance
  • Adjust class weights: class_weight='balanced'
  • Use ROC-AUC which is threshold-independent

  • Start with default parameters as baseline
  • Use RandomizedSearchCV for large spaces
  • Define meaningful search ranges
  • Use cross-validation within tuning (cv=5)
  • Compare tuned vs baseline for improvement

  • Remove steps intentionally to see impact
  • Try different random states for stability
  • Visualize feature importance from trees
  • Compare metrics across datasets
  • Test with different data sizes
  • Document all findings in notebooks
Golden Rule: Failed experiments teach as much as successful ones!
