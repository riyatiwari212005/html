🎯 ATS Microservices: 15 Core Services

1. AGENT11_PIPELINE_SERVICE

Purpose: Core orchestration service that manages and coordinates workflow pipelines within the recruitment AI agent system.

  • Built with FastAPI and Python 3.8+
  • Exposes RESTful APIs on port 8088
  • Comprehensive Swagger documentation included
  • Centralized endpoint for pipeline operations
  • Coordinates multi-step recruitment processes

2. AI_AGENT_CANDIDATE_RECOMMENDATION_SYSTEM

Purpose: Advanced AI-powered matching system that automatically matches candidates with job openings.

  • Leverages Sentence-BERT for semantic understanding
  • Cross-encoder reranker for improved accuracy
  • Structured field matching (skills, experience, education, location)
  • Role-based access control
  • Optional GPU acceleration for 5-10x faster inference
  • Sub-5-second response times

3. AI_AGENT_LOGIN1_Service

Purpose: Critical authentication and user management microservice for the recruitment platform.

  • Handles all authentication operations
  • Role-based access control (admin, user, moderator, guest)
  • JWT token generation and validation
  • Account status validation (enable/disable)
  • MySQL database integration
  • Secure password hashing

4. AI_AGENT1_File_Handler

Purpose: Dedicated file management microservice for document operations.

Key Architectural Characteristics

🔌 Microservices Architecture

Loosely-coupled, independently deployable services that communicate via APIs and message queues.

📦 Docker Containerization

All services containerized for consistency, scalability, and easy deployment across environments.

⚡ Asynchronous Processing

Celery + Redis for background task processing, enabling responsive user interfaces.

🌐 Network Isolation

Shared Docker networks enable inter-service communication while maintaining security boundaries.

🔐 Security

JWT-based authentication, role-based access control, and credential management via Jenkins.

📊 Database Integration

MySQL backend with SQLAlchemy ORM for data persistence and complex queries.

Technology Stack

Framework & Language:

  • Pydantic (data validation)
Database & Caching:

Asynchronous & Messaging:

AI & ML Integration:

🚀 Jenkins CI/CD Pipeline Management

⚙️ Pipeline Architecture

All services follow the same standardized Jenkins pipeline with the following stages:

1️⃣ Guard Branch ├── Validates that only allowed branches trigger builds └── Currently configured: qa branch only 2️⃣ Init ├── Initializes environment variables └── Sets up configurations 3️⃣ Checkout ├── Clones source code from repository └── Fetches latest branch 4️⃣ Build Image ├── Builds Docker image from Dockerfile └── Tags with service name and environment 5️⃣ Stop Old Container ├── Gracefully stops previous container └── Removes associated volumes 6️⃣ Run Container ├── Starts new container with credentials ├── Mounts volumes for logs and files └── Connects to shared network 7️⃣ Post ├── Cleanup operations └── Reports build status

Build Triggers & Configuration

🔄 Build Triggers:
  • Branch Filtering: Only qa branch triggers builds
  • Concurrent Builds: Disabled to prevent simultaneous deployments
  • Early-stage validation: Prevents builds on non-qa branches
stage('Guard Branch') { when { expression { return env.BRANCH_NAME != null } } steps { if (env.BRANCH_NAME != 'qa') { echo "Skipping build: only qa branch is allowed" currentBuild.result = 'NOT_BUILT' } } }

📦 Docker Build & Deployment Process

Build Stage

stage('Build Image') { when { branch 'qa' } steps { sh "docker build -t ${IMAGE_NAME} ." } }

Creates a Docker image from the Dockerfile in the service repository using the image name pattern: ats-<service>-<env>

Container Lifecycle Management

🛑 Stop Old Container

stage('Stop Old Container') { steps { sh """ docker stop ${CONTAINER} 2>/dev/null || true docker rm -f ${CONTAINER} 2>/dev/null || true """ } }
Process: Gracefully stops the running container (if exists), removes the container and associated volumes, and ignores errors if container doesn't exist.

🚀 Run New Container

stage('Run Container') { steps { sh """ docker run -d --name ${CONTAINER} --restart unless-stopped \\ --add-host=host.docker.internal:host-gateway \\ --network ${NETWORK} \\ -p ${APP_PORT}:${APP_PORT} \\ -e APP_ENV=${env.APP_ENV} \\ -e APP_PORT=${APP_PORT} \\ [... environment variables ...] \\ -v ${LOG_PATH}:${LOG_PATH} \\ -v ${FILE_PATH}:${FILE_PATH} \\ ${IMAGE_NAME} """ } }

Key Docker Options

Option Purpose
-d Run in detached mode (background)
--restart unless-stopped Auto-restart container on failure
--add-host=host.docker.internal:host-gateway Docker Desktop compatibility
--network Connects to shared network for inter-service communication
-p Port mapping (host:container)
-e Environment variables (from Jenkins credentials)
-v Volume mounts for logs and files

🔐 Jenkins Credentials & Security Management

Credential Storage Pattern

All sensitive data is stored in Jenkins as secrets and injected at runtime:

withCredentials([ string(credentialsId: 'AI-DB-HOST', variable: 'DB_HOST'), string(credentialsId: 'AI-DB-PORT', variable: 'DB_PORT'), string(credentialsId: 'AI-DB-NAME-QA', variable: 'DB_NAME'), string(credentialsId: 'AI-DB-QA-USER', variable: 'DB_USER'), string(credentialsId: 'AI-DB-QA-PASS', variable: 'DB_PASSWORD'), string(credentialsId: 'AI-JWT-SECRET-KEY', variable: 'JWT_SECRET_KEY'), ]) { // Credentials available as environment variables sh "docker run -e DB_HOST=$DB_HOST ..." }

Required Jenkins Credentials

📊 Database Credentials (Shared across services)

  • AI-DB-HOST - Database host address
  • AI-DB-PORT - Database port (typically 3306)
  • AI-DB-NAME-QA - QA database name
  • AI-DB-USER-QA or AI-DB-QA-USER - Database username
  • AI-DB-PASS-QA or AI-DB-QA-PASS - Database password

🔑 Authentication & Security Credentials

  • AI-JWT-SECRET-KEY - JWT signing key for token generation
  • AI-JWT-ALGO - JWT algorithm (typically HS256)
  • AI-HASHING-ALGO - Password hashing algorithm (bcrypt)

🛡️ Security Best Practices

✓ Implemented Security Measures:
  • All credentials stored in Jenkins secret store
  • Credentials injected as environment variables at runtime
  • JWT-based authentication across services
  • Role-based access control (RBAC) service
  • Password hashing with bcrypt
  • Only QA branch triggers automatic builds
  • Docker network isolation between environments
