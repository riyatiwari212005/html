📊 System Overview

🎯 Mission: A production-grade bot orchestration platform that manages multiple trading bots as containerized services with centralized control, real-time monitoring, and secure credential management.
🤖

HummingBot Manager

Flask-based central control system that orchestrates all bots and manages lifecycle

🐳

Docker Containers

Each bot runs in isolated Docker container for security and portability

🗄️

Database

Secure storage for users, credentials, configs, and transaction history

💬

MQTT Pub/Sub

Real-time messaging for manager ↔ bot communication

📈 Key Features

  • ✅ Centralized Control: Create, start, stop, and monitor multiple bots from one dashboard
  • ✅ Secure Credential Management: Encrypted storage of API keys and secrets
  • ✅ Real-time Monitoring: Live logs, performance metrics, transaction tracking
  • ✅ Dynamic Bot Creation: Spin up new trading bots on demand via API
  • ✅ Configuration as Code: YAML-based bot strategies and parameters
  • ✅ Horizontal Scalability: Support 100s of bots simultaneously
  • ✅ Fault Tolerance: Container restart policies, error recovery, retry logic

Component Breakdown

Purpose: Central orchestration and API endpoint

Responsibilities:

  • Create new bot Docker containers dynamically
  • Start/Stop/Restart bots
  • Inject configurations into containers
  • Monitor bot health and performance
  • Route API requests to appropriate services
  • Manage authentication and authorization

Tech Stack: Flask, Python Docker SDK

What: Each bot runs in its own Docker container

Isolation Benefits:

  • Process isolation (bot crash doesn't affect others)
  • Resource limits (CPU, RAM per bot)
  • Environment isolation (different API keys per bot)
  • Easy scaling (add containers, not code changes)

Container Image: Hummingbot base image + custom strategy

Database: PostgreSQL

Storage Strategy:

  • Encrypted API credentials
  • User account information
  • Bot configurations (YAML stored as JSON)
  • Transaction history
  • Audit logs for compliance

Security: All sensitive data encrypted at-rest and in-transit

Purpose: Real-time bidirectional communication

Topics:

  • bots/{bot_id}/commands - Manager → Bot commands
  • bots/{bot_id}/status - Bot → Manager status updates
  • bots/{bot_id}/logs - Bot → Logging system
  • system/alerts - System-wide notifications

Broker: EMQX or Mosquitto

⚙️ Core Components

1. Bot Orchestration Layer

Responsibility: Manage Docker container lifecycle

Operations:
  • Create new bot container from image
  • Inject config via environment variables
  • Start/Stop containers
  • Monitor resource usage
  • Restart on failure

2. HummingBot Integration

How Bots Execute Trades:

Flow:
  1. Manager generates YAML config from DB
  2. Config injected into container as environment/mounted file
  3. HummingBot reads config on startup
  4. Bot connects to exchange API using credentials from ENV
  5. Bot executes strategy (place orders, monitor, adjust)
  6. Results streamed via MQTT → Logging system

3. Configuration System

Example Bot Config (YAML format):

strategy: pure_market_making exchange: binance trading_pair: BTC-USDT amount: 0.001 # Strategy-specific parameters bid_spread: 0.01 # 1% below market ask_spread: 0.01 # 1% above market order_refresh_time: 30 inventory_target_base_pct: 50 # Credentials (injected via ENV) api_key: ${EXCHANGE_API_KEY} api_secret: ${EXCHANGE_API_SECRET}

4. API Layer (Flask)

Endpoint Method Description Returns
/api/bots/create POST Create new bot Bot ID, Status
/api/bots/{id}/start POST Start bot container Status: started
/api/bots/{id}/stop POST Stop bot container Status: stopped
/api/bots/{id}/logs GET Fetch bot logs (WebSocket for live) Log entries
/api/transactions GET List all transactions Transaction history
/api/credentials/add POST Add exchange credentials Credential ID

Example Create Bot Request:

POST /api/bots/create { "name": "BTC Arbitrage Bot 1", "strategy": "arbitrage", "exchange_primary": "binance", "exchange_secondary": "kucoin", "trading_pair": "BTC-USDT", "amount": 0.001, "min_profitability": 0.003 } Response: { "bot_id": "uuid-123", "status": "created", "docker_id": "abc123def456", "message": "Bot created successfully" }

🎯 Key Points:
• Manager connects to Docker daemon via socket
• Each bot created dynamically (not in compose)
• Core services (DB, MQTT) persistent
• Environment-based configuration for security

Bot Container Lifecycle

1️⃣ Creation: User calls API → Manager creates container
2️⃣ Configuration Injection: Config pushed to container via ENV/volume mount
3️⃣ Startup: Container starts, loads config, connects to exchange
4️⃣ Execution: Strategy runs, streams logs via MQTT
5️⃣ Monitoring: Manager checks health, collects metrics
6️⃣ Termination: User stops bot → Container gracefully shuts down

📌 8 Core MQTT Topics (Detailed)

🎯 Purpose: Manager sends control commands to bot

QoS Level: 1 (at least once delivery)

Supported Commands:
Command Description Parameters
start Start trading strategy config (JSON)
stop Stop trading gracefully reason (string)
restart Restart bot container force (boolean)
pause Pause trading (keep orders) duration (seconds)
resume Resume from pause —
update_config Update strategy parameters config (JSON)
panic_sell Emergency: sell all positions —

Scenario: Manager wants to update bot strategy spreads

Step 2 (T+0.1s): Frontend calls API → /api/bots/bot-123/config
Step 3 (T+0.2s): Manager receives request, validates in DB
Step 4 (T+0.3s): Manager publishes to bots/bot-123/commands
                Payload: {"command": "update_config", "payload": {"bid_spread": 0.015}}
Step 5 (T+0.5s): Bot receives command from MQTT broker
Step 6 (T+0.6s): Bot validates & applies new config
Step 7 (T+5s): Bot publishes status with new config to bots/bot-123/status
Step 8 (T+5.1s): Manager receives status update, updates dashboard

⚠️ Bottlenecks & Improvements

Critical Bottlenecks

1. Single Manager Failure Risk

Problem: If manager crashes, can't start/stop bots
Solution:
  • Run 2-3 manager replicas behind load balancer
  • Use distributed state (shared DB)
  • Health check + auto-restart
2. Database Contention

Problem: 500 bots × frequent writes = high DB load
Solution:
  • Use read replicas for queries
  • Move logs to time-series DB (InfluxDB, TimescaleDB)
  • Cache frequently accessed configs in Redis
  • Batch writes where possible
3. MQTT Broker Scaling Limits

Problem: Single MQTT broker can handle ~1K connections max
Solution:
  • Use EMQX cluster (multi-node)
  • Implement message routing/sharding
  • Consider Kafka for higher throughput
4. Network I/O Between Bot & Exchange

Problem: 500 bots all hitting exchange API → rate limits
Solution:
  • Implement exponential backoff retry logic
  • Use connection pooling
  • Respect API rate limits per exchange
  • Queue requests intelligently
