🏗️ Platform Architecture Overview

What is BRN?
BRN (BRCHAT) is an enterprise AI chatbot platform. Companies build intelligent conversational agents with human fallback, voice support, and external tool calls.
  • 🤖 AI Responses — Queries Neo4j knowledge graph + Anthropic Claude
  • 👤 Human Escalation — HITL operators take over when AI can't help
  • 🎤 Voice Channels — Web mic, WhatsApp, phone calls, ElevenLabs
  • 🔧 External Tools — MCP (Model Context Protocol) for 3rd-party APIs
  • 📊 Multi-Tenant — Complete company-level data isolation
  • ⚡ Real-Time — WebSockets + Redis pub/sub

👤 HITL Service: Human-in-the-Loop

Port: 8109 | FastAPI + WebSockets + MySQL + Redis | Gemini LLM
What is HITL?
HITL = Human-in-the-Loop. When the chatbot can't help, this service:
✅ Puts the user in a queue waiting for a human operator
✅ Notifies available operators in real-time
✅ Operator accepts the chat and talks live to the user
✅ Handles transfers, rejections, notes, resolutions
✅ Tracks every operator's online/offline/busy status
✅ Provides AI reply suggestions powered by Google Gemini

💬 Response Service: Chatbot Engine

Port: 8029 (main) + 8030 (MCP config) | FastAPI + Claude + Neo4j + MCP
What is the Response Service?
The 🧠 BRAIN. Runs as two apps in one container:
• Port 8029 — Main chatbot (messages, responses, streaming, HITL, MCP)
• Port 8030 — MCP config validation and server status checks
  • Receives messages via REST or WebSocket
  • Queries Neo4j knowledge graph (graph + vector search in parallel)
  • Generates answers with Anthropic Claude Sonnet (streaming)
  • Decides when to hand off to a human (HITL)
  • Calls external APIs via MCP (Model Context Protocol)
  • Tracks every token used and every message stored

Knowledge Retrieval — Two Parallel Paths
📐 PATH 1 — Graph Retrieval (Structured)
User question
↓ Entity extraction via Claude Haiku
Entities: ["pricing", "plan A"]
↓ Cypher query on Neo4j:
MATCH (n:__Entity__) WHERE n.id IN [...]
MATCH (n)-[r]-(m) RETURN n, r, m
✅ Returns: nodes + relationships (structured knowledge)
🔍 PATH 2 — Vector Retrieval (Semantic)
User question
↓ Gemini embedding → vector representation
↓ Similarity search in Neo4j vector index
"Find Document nodes closest to question"
✅ Returns: top-K most relevant text chunks
Both results COMBINED → fed into Claude Sonnet → final streaming answer
Fallback / Resilience
If main answer generation fails (Neo4j error, timeout etc.):
→ _generate_fallback_answer_with_haiku() called
→ Best-effort context fetch, then Claude Haiku (faster/cheaper) generates a helpful response
→ User always gets a response — never a blank error

The Training Pipeline (All 12 Steps)
1
API receives request — file + agent_id + file_name + options
2
Create database records — KnowledgeBase (PENDING), Documents, JobFiles
3
Dispatch Celery task — return {job_id, task_id} to client immediately; client polls for progress
4
Celery worker picks up task — updates status to "TRAINING"
5
Load document: URL→Selenium scrape | DOCX→convert to PDF | image_train→high-res OCR | CSV→table load | PDF→direct
6
Chunk text into overlapping segments (progress ~30–40%)
7
Claude LLM graph extraction — each chunk sent to Claude; extracts entities + relationships; tokens recorded (progress ~40–80%)
8
Generate embeddings using Google Gemini (vector representations)
9
Store in Neo4j — all nodes tagged with tenant_id for isolation (progress ~80–90%)
10
Upload to Wasabi S3 — original file backed up, URL stored in DB
11
Mark TRAINED (100%) — chatbot can now query this knowledge
12
If anything fails — status → FAILED, error stored, file still on S3, retry without re-uploading

Bot offers: "Would you like to speak to a human?" — stored in Redis as handoff_offer
3
User accepts → Response Service calls HITL: POST /agent-hitl/handoff/create
4
HITL creates HandoffQueue → PENDING, 30-min timer starts
5
Auto-assignment: find ONLINE operator with capacity, ordered by routing_priority
6
All operators notified via Redis pub/sub + WebSocket /ws/handoff-queue
7
Operator clicks "Accept" → handoff ACCEPTED, current_chats incremented
8
HITL calls Response Service: POST /hitl/unified (event: "accepted")
9
session.handler = "human", welcome message sent to user
10
Live chat: user messages → Response Service → HITL → operator WS
11
Operator resolves → session.handler = "bot", chat count decremented
Flow 3: MCP Tool Call + Elicitation
1
User: "Book me a flight to Dubai next Friday"
2
Claude decides: needs "flights_search_available" tool
3
MCP Gateway parses: server="flights", tool="search_available"
4
Auth Manager gets token for "flights" server (bearer / OAuth / JWT)
5
Tool needs more info → ElicitationPendingError → form stored in Redis
6
User receives elicitation form: {type:"elicitation", fields:[{name:"class",label:"Seat class?"}]}
7
User submits: POST /elicitation/{session_id}/respond
8
Tool call RESUMED with filled data → executes → returns results
9
Claude generates final answer: "Found 3 flights to Dubai on May 1..."
